# Plan: Add relevanceSummary Field

## Problem

The assessment prompt and Zod schema (`assessResultSchema`) include a `relevanceSummary` field (20-25 word high-level summary of why a story is relevant), but the field is **never saved to the database**. It's generated by the LLM and discarded in `assessStory()`.

On the public home page, the `StoryCard` component uses a deterministic coin flip (`shouldShowRelevance`) to sometimes show the first relevance reason (extracted from `relevanceReasons`) instead of a quote. This is a poor substitute — relevance reasons are detailed bullet points, not concise summaries.

## Solution

Persist `relevanceSummary` to the database, expose it in admin for editing, and use it on public story cards instead of the first relevance reason.

## Changes

### Phase 1: Database Migration

1. **`server/prisma/schema.prisma`** — Add `relevanceSummary String? @map("relevance_summary")` to the Story model (after `relevanceReasons`).

2. **Generate SQL migration** — Run `npx prisma migrate dev --create-only --name add_relevance_summary` via npm script, then follow the project's SQL-first migration workflow (generate SQL, user applies in pgAdmin, mark resolved).

### Phase 2: Server — Persist the Field

3. **`server/src/services/analysis.ts`** (~line 245) — In the `assessStory()` function, add `relevanceSummary: parsed.relevanceSummary || null` to the `prisma.story.update()` data object.

4. **`server/src/services/story.ts`** — Add `relevanceSummary` to the `PUBLIC_STORY_SELECT` object so it's returned in public API responses.

5. **`server/src/routes/admin/stories.ts`** — Verify `updateStorySchema` allows `relevanceSummary` (it likely uses `Partial<Story>` or a permissive schema — confirm and add if needed).

### Phase 3: Shared Types

6. **`shared/types/index.ts`** — Add `relevanceSummary: string | null` to the `Story` interface (after `relevanceReasons`).

### Phase 4: Admin UI

7. **`client/src/components/admin/StoryEditForm.tsx`** — Add a textarea field for `relevanceSummary` in the analysis section (near `marketingBlurb` or `relevanceReasons`).

### Phase 5: Public Story Cards

8. **`client/src/components/StoryCard.tsx`** — Replace the `getFirstRelevanceReason()` usage with `story.relevanceSummary`. The logic becomes:
   - For `featured` and `horizontal` variants: if `shouldShowRelevance(story.id)` is true AND `story.relevanceSummary` exists, show the relevance summary instead of quote/summary.
   - Remove the `getFirstRelevanceReason()` function (no longer needed).
   - Keep `shouldShowRelevance()` — it's still useful for the coin flip between relevanceSummary and quote.

### Phase 6: Cleanup

9. **Remove unused imports** — `parsePoints`, `stripMarkdown`, `stripPrefix`, `limitSentences` if no longer used in StoryCard.

## Risks

- **Existing stories** won't have `relevanceSummary` populated until re-assessed. The coin flip + null check handles this gracefully — stories without the field fall back to quote/summary as before.
- **Migration** is a simple nullable column add — no data loss risk.
